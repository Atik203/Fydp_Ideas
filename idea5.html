💡 1. Core Idea

We propose StatVerify-Agent: an LLM-based autonomous agent that reads statistical claims in a scientific paper's Results section, identifies the corresponding supplementary dataset, writes Python code to reproduce the reported statistical test (t-test, ANOVA, chi-squared, etc.), executes it in a sandboxed environment, and reports whether the computed result matches the paper's stated conclusion. This shifts verification from semantic plausibility to computational reproducibility.

🎯 Core Research Contribution

The first end-to-end agentic pipeline for automated statistical reproducibility auditing of scientific papers. Secondary contribution: a taxonomy of irreproducibility causes derived from systematic failure analysis — itself a publishable finding for the science policy community.

🔍 2. Problem Statement

2.1 The Reproducibility Crisis

An estimated 70% of researchers have tried and failed to reproduce another scientist's experiments (Baker, 2016, Nature). Nuijten et al. (2016) found that 49.6% of psychology papers contained at least one statistical reporting error, with 12.8% containing errors that changed the significance conclusion. These errors persist because peer reviewers lack the time and tools to re-run analyses from scratch.

2.2 Why This Task Requires an Agent

Unlike other ideas in this proposal, this task genuinely requires an agentic architecture. The workflow is inherently multi-step with feedback loops:

Planning Required

"I need to run a paired t-test on columns X and Y, apply Bonferroni correction, and compare with the reported p=0.003." This multi-step plan cannot be expressed as a single prompt.

Tool Use Required

Actual code execution in a Python REPL is necessary — the answer cannot be reasoned about in text alone. Scipy, pandas, and statsmodels must be called and results read.

Self-Correction Required

If the code errors (wrong column name, type mismatch), the agent must parse the traceback, identify the fix, and re-run — a classic ReAct loop that cannot be handled by a static pipeline.

Verdict Generation Required

The agent must compare computed vs. reported p-values, account for rounding, and generate a structured verdict (MATCH / MISMATCH / INCONCLUSIVE) with a complete evidence trace.

⚠ Explicit Research Gap

No system implements an autonomous end-to-end pipeline: reading a paper's statistical claim → identifying the relevant dataset → generating correct statistical test code → executing it → comparing the result to the reported value. The gap is in agentic orchestration of these four steps as a self-correcting loop.

🏗️ 4. StatVerify-Agent Architecture

Component 1 — Claim Extractor

Scans Results/Discussion sections and outputs structured claim objects:

  • claim_text: "Treatment group showed significantly lower scores (t(48)=3.21, p=0.002)"
  • test_type: independent t-test
  • reported_statistic: t=3.21, df=48, p=0.002
  • data_reference: Table 2 / Supplementary File S1

Component 2 — Plan-and-Solve ReAct Agent (Core)

Implemented via LangGraph state machine with conditional edges for error recovery (max 3 retries):

  1. Plan: "Load S1.csv, identify treatment/control columns, run scipy.stats.ttest_ind"
  2. Act: Generate Python code
  3. Observe: Execute in Dockerized sandbox; parse output or traceback
  4. Reflect: If error: "Column was 'Treatment_Score', not 'treatment'. Fix and retry."
  5. Loop (max 3 iterations; returns INCONCLUSIVE if unresolved)

Component 3 — Verdict Generator

MATCH

|pcomputed − preported| < ε (ε=0.005 default). Paper's statistical claim is reproducible.

MISMATCH

Significant divergence detected. Full code trace, both p-values, and discrepancy magnitude reported.

INCONCLUSIVE

Max retries exceeded or data format unresolvable. Reason code returned (categorised for the failure taxonomy).

SCOPE ERROR

Claim identified but data not available in supplementary files. Common case; reported as data availability failure.

🔬 5. Methodology

5.1 Data Collection (Primary Bottleneck)

Source: PLOS ONE (CC-BY licensed, mandates data availability). Select 100 papers where: (a) Results section contains APA-style statistical reporting, and (b) supplementary data is a downloadable CSV/XLSX file (bypassing PDF extraction entirely). Manual gold annotation: human annotator runs each analysis independently.

Error injection: For 50% of the test set, deliberately introduce one of four error types: (1) wrong p-value by ±20%, (2) incorrect test type, (3) unreported data filtering, (4) n manipulation. Measures error detection sensitivity.

⚠ Risks & Mitigations
  • Data curation bottleneck (HIGH RISK): Manual gold annotation of 100 papers' statistical analyses takes 2–3 months. Mitigation: Start curation in Month 1 in parallel with system design. Consider splitting annotation across 2 team members if possible.
  • PDF extraction (BYPASSED): Only papers with downloadable CSV supplementary data are included. This reduces corpus size but eliminates the PDF parsing problem entirely.
  • Novelty concerns (REAL RISK): ReAct agents are not new. Mitigation: Frame as the first domain-specific deployment for statistical auditing; emphasise the failure taxonomy as a standalone contribution; target SDP@EMNLP workshop as first venue.
  • Code generation correctness: LLMs may generate syntactically correct but semantically wrong code. Mitigation: Add explicit test-justification step before code generation; human-evaluate 50-sample subset.
