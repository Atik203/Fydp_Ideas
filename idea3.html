💡 1. Core Idea

LLMs treat knowledge as timeless, blending facts from different eras into incoherent answers. Standard RAG compounds this by retrieving the "most semantically similar" text regardless of temporal validity. We propose Chrono-RAG — a retrieval-augmented generation framework that routes queries through a Temporal Knowledge Graph (TKG), where every fact is stored as a time-bounded quadruple (Subject, Predicate, Object, [t_start, t_end]). The system enforces chronological isolation: only facts valid at the queried time point are injected into the LLM context, and conflicting temporal facts are explicitly surfaced as version histories.

🎯 Core Research Contribution

Chrono-RAG: a three-layer retrieval architecture treating temporal validity as a first-class constraint in both knowledge representation (TKG schema with time-bounded quadruples) and retrieval algorithm (temporal filtering before semantic ranking). Neither component alone constitutes the contribution — the novelty is in their unified integration.

🔍 2. Problem Statement

2.1 The Temporal Hallucination Problem

Knowledge evolves. The WHO reversed its mask guidance in June 2020. A country's head of state changes every election cycle. Drug dosage guidelines update annually. LLMs trained on static snapshots conflate these temporal versions into a "probability-weighted average" that may never have been true at any specific point in time.

Kasai et al. (2023) demonstrated GPT-4's accuracy on time-sensitive questions drops 18–25% compared to static factual questions. Liska et al. (2022) showed even fine-tuned models struggle to distinguish "X was true in 2019" from "X is true now." This is not a parametric knowledge problem — it is a representation problem: LLMs have no internal clock and no mechanism for temporal isolation.

2.2 Why Standard RAG Fails

Dense passage retrieval (DPR) ranks documents by semantic similarity, not temporal validity. A highly-cited 2018 treatment protocol outscores a 2024 update with fewer citations. The retriever cannot understand that the older document has been superseded. Even appending retrieval timestamps to the prompt is insufficient — the LLM is not trained to use them as hard temporal constraints.

2.3 Engineering Complexity

Challenge A: Temporal Schema Design

Representing facts as time-bounded quadruples and resolving overlapping or open-ended temporal intervals (e.g., facts that are still valid: [2020, ∞]).

Challenge B: Query Temporal Resolution

Extracting temporal constraints from natural language queries — distinguishing explicit ("in 2021"), implicit ("during COVID"), and absent temporal references.

Challenge C: Conflict Detection

Identifying when two facts about the same entity conflict across time periods and generating a coherent temporal evolution narrative rather than blending them.

Challenge D: TKG Coverage

Pre-built TKGs (YAGO3, Wikidata) have variable temporal annotation density across domains — sparse in niche domains, dense in politics/science/medicine.

⚠ Explicit Research Gap

No RAG system treats temporal validity as a first-class constraint in both the knowledge representation schema and the retrieval algorithm. Existing systems are temporally blind by architectural design — they can retrieve about time but cannot enforce temporal isolation during retrieval.

🏗️ 4. Chrono-RAG Architecture (3 Layers)

Layer 1 — Temporal Knowledge Graph

Facts stored as time-bounded quadruples: (entity, relation, entity, [t_start, t_end]). Examples:

  • (WHO, recommends_against, general_mask_use, [2020-01, 2020-06])
  • (WHO, recommends, general_mask_use, [2020-06, ∞])
  • (Joe_Biden, president_of, USA, [2021-01, 2025-01])

Sources used (pre-built — no TKG construction from scratch):

YAGO3
~3M temporal facts
Wikipedia-derived; time intervals on entities
Wikidata Temporal
~50M qualifiers
point-in-time, start-time, end-time properties
T-REx
~6.2M triples
Wikipedia-aligned temporal relations

Layer 2 — Temporal Query Resolution

A fine-tuned Temporal Constraint Extractor (RoBERTa-base on TempQuestions) identifies:

  • Explicit: "in 2021" → [2021-01-01, 2021-12-31]
  • Implicit: "during the pandemic" → [2020-03, 2022-05] (mapped via event ontology)
  • Absent: Default to current date (open temporal query)

The extracted window [t_query_start, t_query_end] filters the TKG before any semantic retrieval occurs — temporal isolation is enforced at the data access layer, not at the generation layer.

Layer 3 — Filtered Traversal + Generation

The temporally filtered subgraph is serialized as structured text (entity–relation–entity chains) and injected into the LLM prompt. When conflicting facts exist across adjacent time windows, the system generates an explicit temporal evolution narrative rather than blending them.

Gain on Static Facts

🎓 8. Why This Meets Q1 Standards

Production-Critical Problem

Temporal hallucination is a top-3 reason enterprises distrust LLM deployments in legal, medical, and financial contexts. This directly addresses a deployment blocker.

Novel Unified Architecture

Temporal query resolution + TKG-filtered retrieval + evolution-aware generation has no published precedent as a unified system. Each component is independently evaluable.

⚠ Risks & Mitigations
  • TKG construction (CRITICAL MISUNDERSTANDING RISK): Building a TKG from scratch is a PhD-level project. Mitigation: We use only pre-existing TKGs (YAGO3, Wikidata). The contribution is the retrieval architecture, not the knowledge graph itself.
  • Implicit temporal resolution: "During COVID" → ambiguous date mapping. Mitigation: Restrict main experiments to explicit temporal queries; report implicit resolution as a separate harder sub-task with honest numbers.
  • TKG coverage gaps: Sparse temporal annotations in niche domains. Mitigation: Restrict evaluation to densely annotated domains (politics, medicine, science) where YAGO3/Wikidata coverage is verified.
  • Reviewer concern "just a date filter": A skeptical reviewer may argue the contribution is trivial. Mitigation: Emphasise the novelty of temporal constraint extraction at the query level, and show with Condition C (RAG + metadata) that simply appending dates does not solve the problem.
ℹ Agentic AI Relevance

An agentic architecture is optionally beneficial here as an extension. A planning agent could decompose complex temporal queries into sub-queries (e.g., "First, determine when policy X changed; then retrieve the version valid at that time"). This is a natural Phase 2 extension for a PhD, not a core FYDP requirement. The base Chrono-RAG pipeline works without agents.
